# Plan: Add Custom Local Models to Keeper from Board UI

## Goal
Allow admins to add arbitrary small local Ollama models (e.g. `phi4-mini`, `gemma3:1b`) to the Keeper permitted list directly from the **Keeper Settings** page (`/settings/keeper`) in the React Board UI. Today the list is hardcoded in Python and requires a code change + container restart to extend.

---

## Architecture Decision

Store the dynamic permitted-model list in the existing **`system_settings`** table (key=`local_permitted_models`, scope=`system`, value=JSON dict `{gen:[...], embed:[...], stt:[...], tts:[...]}`). 

**Why not a new table?**
- The `SystemSetting` table already supports JSON values, has CRUD endpoints (`/v1/settings/{key}`), has a 30-second TTL cache via `dynamic_config.py`, and is wired into the Board UI via `useSettings` / `useUpdateSetting` hooks.
- A new table would require a migration, new router endpoints, and new React hooks for minimal gain. The SystemSetting pattern is already used for `keeper_timeout_seconds`, `agent_spawn_timeout_seconds`, etc.

**Fallback behavior:** If the setting does not exist in the DB, the backend falls back to the current hardcoded defaults. This ensures the feature works immediately on existing deployments without requiring a migration or manual seeding.

---

## Files to Change

### 1. Backend — `isli-core/src/isli_core/routers/model_management.py`

**Changes:**
1. Rename the hardcoded dict to `DEFAULT_PERMITTED_MODELS`.
2. Add helper `async def _get_permitted_models(db: AsyncSession) -> dict[str, list[str]]`:
   - Queries `SystemSetting` key=`local_permitted_models`, scope=`system`.
   - Returns the JSON value merged over defaults (so missing slots still have defaults).
   - Uses the existing `get_setting()` helper from `dynamic_config` (or inline query) for the 30s cache benefit.
3. Update `GET /status` to call `_get_permitted_models` instead of reading the hardcoded dict.
4. Update `POST /activate`, `POST /remove`, `POST /pull` to call `_get_permitted_models` for validation.
5. Add **`POST /model-management/permitted`** (body: `{slot, model_name}`):
   - Reads current permitted models.
   - Appends `model_name` to the slot list if not already present.
   - Upserts `SystemSetting` key=`local_permitted_models`.
   - Calls `invalidate_cache(key="local_permitted_models")`.
6. Add **`DELETE /model-management/permitted/{slot}/{model_name}`**:
   - Reads current permitted models.
   - Removes `model_name` from the slot list.
   - Saves back to `SystemSetting`.
   - Calls `invalidate_cache`.
   - Returns 404 if the model was not in that slot.

**Validation rules:**
- `slot` must be one of `gen`, `embed`, `stt`, `tts`.
- `model_name` must be non-empty, max 128 chars (same as `permitted_models.model_id`).
- No Ollama-side validation on add — the model name is just a string in the list. If the user types an invalid Ollama tag, `pull` will fail later with a clear Ollama error.

**Active-session guard:** The new `POST` and `DELETE` endpoints **do not** need the active-session block. Changing the permitted list is safe while agents are running; only `activate`/`remove`/`pull` mutate runtime state.

### 2. Backend — `isli-core/src/isli_core/startup/settings_seed.py`

**Changes:**
- Add one entry to `DEFAULT_SETTINGS`:
  ```python
  {"key": "local_permitted_models", "scope": "system", "value": {
      "gen": ["qwen3:1.7b", "qwen3:4b", "mistral:7b", "qwen2.5-coder:1.5b"],
      "embed": ["nomic-embed-text", "mxbai-embed-large"],
      "stt": ["whisper-tiny", "whisper-base", "whisper-small", "whisper-medium"],
      "tts": ["piper-en-us-lessac-medium", "piper-en-us-amy-medium"],
  }, "description": "Permitted local models per slot for Keeper and Audio services."}
  ```
- The seeder runs on Core startup and uses `INSERT ... ON CONFLICT DO NOTHING`, so existing deployments are not overwritten.

### 3. Frontend — `isli-board/src/components/LocalModelSettings.tsx`

**Changes:**
1. **New state:** `const [newModelInputs, setNewModelInputs] = useState<Record<string, string>>({})` — per-slot input text.
2. **New handlers:**
   - `handleAddPermitted(slot, modelName)` → `postJSON('/v1/model-management/permitted', { slot, model_name: modelName.trim() })` → `fetchStatus()`.
   - `handleRemovePermitted(slot, modelName)` → `deleteJSON(`/v1/model-management/permitted/${slot}/${encodeURIComponent(modelName)}`)` → `fetchStatus()`.
3. **UI additions in `ModelModule` (gen/embed) and `AudioModule` (stt/tts):**
   - At the bottom of each slot's model list, add a compact input row:
     ```
     [ model_name input ] [ + Add Model ]
     ```
   - On the **right side** of each model card (next to the existing action buttons), add a small `X` icon button for **"Remove from permitted list"**. 
     - Only shown when the model is **not active** (the active model cannot be removed from the list; it must be deactivated first via switching to another model).
     - Clicking it calls `handleRemovePermitted` immediately (no heavy confirmation needed; re-adding is trivial).
   - The existing **trash icon** continues to mean **"Remove from disk"** (delete weights from Ollama/Audio). This distinction is important.
4. **Busy states:** Extend the `loading` string format to include `add-permitted:${slot}:${model}` and `remove-permitted:${slot}:${model}` so spinners appear on the relevant buttons.

**UX notes:**
- The input placeholder: `"ollama tag, e.g. gemma3:1b"`.
- Audio models remain hardcoded on the audio service side, but the UI will still allow adding them to the list. If a user adds an invalid audio model, the audio service's `/admin/activate` and `/admin/pull` will reject it with their existing validation. This is acceptable — the list in Core is just a "wishlist", and the downstream service validates feasibility.

### 4. Frontend — `isli-board/src/hooks/useSettings.ts` (optional)

**No changes required.** The new endpoints are in `model-management`, not `/v1/settings`, so we add direct `postJSON` / `deleteJSON` calls inside `LocalModelSettings.tsx` rather than going through the settings hooks. This keeps the component self-contained and matches the existing pattern in `LocalModelSettings` (which already calls `postJSON('/v1/model-management/...')` directly).

## Data Flow

```
User types "gemma3:1b" + clicks "Add Model" (gen slot)
  ↓
Board UI: POST /api/v1/model-management/permitted  {slot:"gen", model_name:"gemma3:1b"}
  ↓
Core router (model_management.py):
  - Read SystemSetting key=local_permitted_models
  - Append "gemma3:1b" to gen list
  - Upsert SystemSetting
  - invalidate_cache(key="local_permitted_models")
  - Return updated permitted dict
  ↓
Board UI: refreshes status via GET /v1/model-management/status
  ↓
New card appears for "gemma3:1b" with "Pull" button
  ↓
User clicks "Pull"
  → POST /v1/model-management/pull (existing flow) → Keeper /admin/pull → Ollama /api/pull
```

---

## Rollback / Safety

- **DB is the source of truth.** Deleting the `local_permitted_models` row from `system_settings` instantly reverts to the hardcoded defaults.
- **No migration required.** The feature is fully backward-compatible; absent the setting, the backend behaves exactly as it does today.
- **Seeder is non-destructive.** `ON CONFLICT DO NOTHING` means existing rows are never overwritten on startup.

## Testing Checklist (for verification)

1. **Backend:**
   - `GET /v1/model-management/status` returns the default list when `local_permitted_models` is absent.
   - `POST /v1/model-management/permitted` adds a model to a slot.
   - `DELETE /v1/model-management/permitted/{slot}/{model}` removes it.
   - `POST /v1/model-management/activate` rejects a model that was removed from the list.
   - `POST /v1/model-management/pull` accepts a newly added model.

2. **Frontend:**
   - Input field appears at the bottom of each module.
   - Adding a model shows a new card with "Pull" button.
   - Removing a model from the list hides the card.
   - Existing activate/remove/pull workflows continue to work.
   - The active model does not show a "Remove from list" button.

3. **Integration:**
   - After adding `gemma3:1b` and pulling, Keeper's Ollama client downloads it successfully.
   - After activating `gemma3:1b`, the Keeper dashboard shows it as the active gen model.

## Open Questions (none blocking)

1. Should we add a size-limit warning in the UI (e.g. "Recommended: < 8B parameters")? 
   - **Decision:** Add a helper text under the input: "Small local models only. Large models may exhaust container memory." No hard enforcement.
2. Should audio slots also be editable?
   - **Decision:** Yes, the API supports all four slots. The UI renders the same add/remove affordances for all modules. Audio service-side validation will catch invalid audio model names at pull/activate time.
